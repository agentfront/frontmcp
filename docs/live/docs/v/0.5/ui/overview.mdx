## Why Use FrontMCP UI?

When your MCP tools return data, AI platforms typically display it as raw text or JSON. FrontMCP UI lets you create visual widgets that make your tool outputs more useful and engaging.

      "temperature": 18,
      "conditions": "Foggy",
      "humidity": 75

## Platform Support

FrontMCP UI adapts to different AI platform capabilities:

| Platform   | Network Access | External Scripts | Widget Display          |
| ---------- | -------------- | ---------------- | ----------------------- |
| **OpenAI** | Full           | CDN allowed      | Inline, Fullscreen, PiP |
| **Claude** | Blocked        | Inline only      | Artifacts               |
| **Gemini** | Limited        | Inline preferred | Basic                   |

The library automatically detects the platform and adjusts rendering accordingly.

## Tool UI metadata at a glance

Every `@Tool` exposes a `ui` block that describes how the widget should render across hosts. Configure these settings once and the renderer keeps your UX consistent everywhere.

| Setting            | Controls                                                             | Use it when                                                                 |
| ------------------ | -------------------------------------------------------------------- | --------------------------------------------------------------------------- |
| `displayMode`      | Whether the widget renders inline, fullscreen, or picture-in-picture | Highlight dashboards inline or promote immersive flows into fullscreen/PiP. |
| `servingMode`      | How the HTML is delivered (`inline`, `static`, `hybrid`, URL-based)  | Precompile heavy React widgets or reuse shells across multiple responses.   |
| `widgetAccessible` | Allows widgets to call tools through the MCP Bridge                  | Create follow-up actions (retry, escalate, schedule) directly from the UI.  |
| `csp`              | Allowed domains for scripts, images, fonts, and network calls        | Restrict remote dependencies per platform and satisfy Claudeâ€™s strict CSP.  |
| `hydrate`          | Ships the React runtime for client-side interactivity                | Power forms, tabs, and charts that need local state or event handlers.      |

<Tip>
Pair metadata with a concise `widgetDescription` so clients can preview the UI before rendering it, just like Mintlify recommends leading with the most important context.
</Tip>

<Warning>
Limit `csp.resourceDomains` and `csp.connectDomains` to only the hosts your widget truly needsâ€”broad allowlists can leak data or be blocked entirely by conservative clients.
</Warning>

## OpenAI App SDK resource widgets

Need a richer layout than a single card? `@frontmcp/ui` ships `resourceWidget`, `resourceList`, `resourceItem`, `codePreview`, and `imagePreview` helpers so your responses match OpenAI Apps SDK expectations without manual HTML.

```typescript
import { resourceWidget } from '@frontmcp/ui';

const widget = resourceWidget({
  type: 'document',
  title: ctx.output.title,
  description: ctx.output.summary,
  status: ctx.output.archived
    ? { label: 'Archived', variant: 'secondary' }
    : { label: 'Active', variant: 'success' },
  meta: {
    author: ctx.output.author,
    updatedAt: ctx.output.updatedAt,
    tags: ctx.output.tags,
  },
  actions: [
    {
      label: 'Open in CRM',
      icon: 'â†—ï¸',
      href: `https://crm.example.com/notes/${ctx.output.id}`,
      variant: 'primary',
    },
    {
      label: 'Sync',
      icon: 'â™»ï¸',
      htmx: {
        post: `/api/notes/${ctx.output.id}/sync`,
        target: '#note-sync',
        swap: 'innerHTML',
      },
    },
  ],
});
```

<Tip>
Combine `resourceList` with `showLoadMore` and HTMX actions to paginate large datasets without writing any client-side JavaScript.
</Tip>

<Warning>
Escape every user-provided field via `escapeHtml` or `ctx.helpers.escapeHtml` before injecting it into widget helpersâ€”App SDK platforms will reject or sanitize unsafe markup.
</Warning>

See `/docs/ui/components/resource-widgets` for the full API surface plus preview helpers.

## Quick Example

Add a visual widget to your tool in just a few lines:

```typescript title="src/tools/weather.tool.ts"
import { Tool, ToolContext } from '@frontmcp/sdk';
import { card, badge } from '@frontmcp/ui';
import { z } from 'zod';

@Tool({
  name: 'get_weather',
  description: 'Get current weather for a location',
  inputSchema: {
    location: z.string().describe('City name'),
  },
  ui: {
    template: (ctx) => card(`
      <div class="text-4xl mb-4">ğŸŒ¤ï¸</div>
      <div class="text-2xl font-bold">${ctx.output.temperature}Â°C</div>
      <div class="text-gray-600">${ctx.output.conditions}</div>
    `, {
      title: ctx.output.location,
      subtitle: 'Current Weather',
    }),
  },
})
export class GetWeatherTool extends ToolContext {
  async execute(input: { location: string }) {
    // Fetch weather data...
    return {
      location: input.location,
      temperature: 18,
      conditions: 'Partly Cloudy',
    };
  }
}
```
